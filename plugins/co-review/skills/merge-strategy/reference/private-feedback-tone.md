# 私聊反馈语气与边界规范

供 `merge-strategy` skill 在 `with_private_feedback=true` 时严格遵循。

## 核心原则：中立 + 建设性

私聊反馈不是"批评信"，也不是"表扬信"。
它是一份**基于客观证据的、建设性的、针对个人的工作总结**。

---

## 三大边界（绝对不可越界）

### 边界 1：不评判态度

❌ 禁止表述：
- "你这次工作很努力"（评判态度）
- "你显得有点马虎"（评判态度）
- "你应该更细心"（评判态度）
- "你的代码质量在退步/进步"（评判变化趋势）

✅ 允许表述：
- "你新增的 6 个文件中有 2 个有对应测试"（事实）
- "src/notification/email.ts 中有 5 处 TODO 标记"（事实）
- "建议在合并前完成或转化为追踪 issue"（建议）

### 边界 2：不做横向对比

❌ 禁止表述：
- "你的完成度比团队平均水平高/低"
- "和 Alice 相比，你的协作意识更强/更弱"
- "团队中 80% 的人都做了 X，你没做"
- 任何包含其他作者名字、统计、对比的内容

✅ 允许表述：
- "你本次改动的测试同步率为 50%"（仅自己的数据）
- "在你修改的 src/auth/UserService.java 中..."（仅自己的内容）

**实施方式**：
- 反馈文件中**禁止出现其他作者的姓名、邮箱、统计数据**
- 唯一的例外：在"与他人的协调点"中，用"另一位开发者"的措辞（不指名道姓）

### 边界 3：不出现对其他作者的评价

❌ 禁止表述（即使是正面的）：
- "Alice 的代码质量更高"
- "团队中其他人都遵守了边界"
- "建议你向 Bob 学习"（暗含对 Bob 的评价）

✅ 允许表述：
- "建议参考 .sop/lessons.md 中的'IDOR 检查清单'教训"（指向文档，不指向人）
- "建议与改动 types/user.ts 第 12-18 行的另一位开发者协调"（不指名道姓）

## 反馈结构模板（按章节）

### 章节 1：贡献概况
**仅事实，不评判**。

✅ 示例：
```
- Commits：12 个
- 代码改动：+854/-142 行，18 个文件
- 主要模块：auth/
- 时间跨度：2026-04-01 ~ 2026-04-25
```

### 章节 2：做得到位的地方
**客观事实，不煽动表扬**。

风格：
- 描述"什么事完成了"，而不是"你做得好"
- 用"是"句式，避免形容词

❌ 不接受：
> "你的代码质量很高，工作非常细致"

✅ 接受：
> "你的工作集中在 auth/ 模块，与 .sop/plan.md 的分工完全一致，没有越界提交。"
> "你修改的所有源文件都有对应的测试文件。"
> "你在引入 UserRepository.findById 新签名时，主动通知了所有调用方。"

### 章节 3：可以改进的地方
**基于客观证据的具体行动项**。

每条必须包含：
1. **事实**（看到了什么）
2. **位置**（具体到 file:line 或 commit hash）
3. **建议**（可执行的下一步）
4. **背景**（如果引用历史教训或考古结论）

❌ 不接受：
> "你的代码完成度有待提高，建议加强测试。"

✅ 接受：
```
- 在 src/notification/email.ts 中有 5 处 TODO 标记需要解决
  - 具体位置：
    - email.ts:34 — `// TODO: implement retry mechanism`
    - email.ts:87 — `// FIXME: handle rate limit edge case`
    - ...
  - 建议：在合并前完成或转化为追踪 issue
```

### 章节 4：与他人的协调点
**仅描述需要协调的事实，禁止评价他人**。

风格：
- 用"另一位开发者"代替具体姓名
- 仅描述需要协调的内容，不评论他人的工作方式

❌ 不接受：
> "Alice 改了你正在改的文件，你需要去找她沟通"

✅ 接受：
```
- types/user.ts 第 12-18 行：你和另一位开发者都修改了这个范围
  - 建议：合并前与对方共同确认无逻辑冲突
```

### 章节 5：历史经验参考
**从 lessons.md 中筛选与该作者本次工作相关的经验**。

风格：
- 提供 FYI 信息，不暗示"你应该已经知道"
- 引用具体的教训编号或任务 ID

✅ 示例：
```
以下历史经验与你本次的工作相关，供参考：

- 「IDOR 检查清单」（任务 b8e2f4a1）— 与你本次的 controller 改动相关
- 「异步通知幂等性」（任务 a3f8c1d2）— 若未来涉及通知系统的进一步改动可参考
```

## 强制声明（每个文件的头尾必须包含）

### 文件头声明
```markdown
> ⚠️ 本文件由 git 数据自动生成，仅作为个人参考。
> ⚠️ 文件不应在团队群组公开传播。
> ⚠️ 反馈以中立 + 建设性为基调，不代表对你工作的全面评价。
```

### 文件尾声明
```markdown
---

> 本反馈基于 git 数据自动生成，不代表对你工作的全面评价。
> 如有疑问请与团队 lead 当面讨论。
> 本文件不应在团队群组公开传播。
```

## 自检清单（生成每份反馈前必须自查）

- [ ] 文件中是否出现了其他开发者的姓名/邮箱？（必须没有）
- [ ] 是否出现了横向对比？（必须没有）
- [ ] 是否有"你应该..."、"你需要..."这种命令式语气？（应改为"建议..."）
- [ ] 是否有形容词评判（"细致"、"马虎"、"优秀"）？（必须没有）
- [ ] 每条改进建议是否都有 file:line 或 commit 证据？（必须有）
- [ ] 文件头尾的三条警告是否齐全？（必须齐全）
- [ ] 历史经验引用是否仅指向文档/任务 ID（不指向人）？（必须仅文档）

## 极端情况处理

### 情况 1：该作者的改动完美无问题
反馈仍要生成，章节 3 写："本次未发现需要改进的项。"
不要为了写满而编造问题。

### 情况 2：该作者完成度极低（🔴）
仍按中立 + 建设性基调，不上升到"批评"。
具体行动项应详细列出每个 TODO/空实现，让该作者有清晰的修复路径。

### 情况 3：该作者违反了考古红线
仍以建议方式表述，不用"违规"、"错误"等评判语气。
明确说明：
- 红线编号
- 红线内容
- 历史背景（为什么这是红线）
- 修复建议

### 情况 4：作者名包含特殊字符（如中文）
文件名按 sanitize 规则处理（拼音 / 转 unicode）。
文件内容中保留原始名（用于打招呼），但 frontmatter 中用 sanitized 版本。

---

## 输出文件命名

`.team-scope/private/<sanitized-name>.md`

清理规则：
- 转小写
- 空格替换为下划线
- 保留 a-z、0-9、_、-
- 中文/特殊字符 → 转拼音 或 替换为 `_`
- 多个连续下划线合并为一个

例：
- `Alice Wong` → `alice_wong.md`
- `张三` → `zhang_san.md` 或 `zhangsan.md`
- `Bob O'Brien` → `bob_obrien.md`

冲突处理：若多人 sanitized 后同名，附加邮箱 local-part 区分：
- `alice_wong.md`
- `alice_wong_alice2.md`
