# 维度化评分规则

供 `merge-strategy` skill 在 `with_scores=true` 时参考。

## 设计原则

1. **维度化，不综合**：4 个维度独立打 🟢/🟡/🔴，**不给单一总分**
2. **证据驱动**：每个评分必须附具体证据（文件、行号、commit）
3. **基于客观信号**：不评判态度、能力、协作风格
4. **可申诉**：开发者看到评分应该能理解原因，并基于证据进行讨论

## 维度 1：完成度（Completion）

直接映射 `completion-analyzer` 的判定结果：

| 评分 | 条件（来自 completion.md）|
|------|------------------------|
| 🟢 高 | TODO=0 + 测试同步率 ≥ 70% + 调试遗留=0 + 空实现=0 |
| 🟡 中 | 不满足"高"条件，但 TODO≤5 + 测试同步率 30-70% + 调试遗留≤1 + 空实现≤1 |
| 🔴 低 | 不满足"中"条件 |

**特殊规则**：
- critical 调试遗留（debugger; / pdb.set_trace()）→ 强制 🔴
- 空实现是 public API → 强制 🔴

## 维度 2：协作健康（Collaboration）

基于 `risks.md` 中该作者参与的风险：

| 评分 | 条件 |
|------|------|
| 🟢 高 | 0 条 🔴 风险 + 接口变更已通知所有调用方 |
| 🟡 中 | 1 条 🔴 风险，或多条 🟡 风险，或接口变更未完全同步 |
| 🔴 低 | ≥ 2 条 🔴 风险 |

**特殊规则**：
- 任何 ⛔ critical 风险 → 不评分（团队级问题，不归因到个人）

## 维度 3：边界遵守（Boundary Compliance）

基于 `contributions.md` 中的越界提交统计：

| 评分 | 条件 |
|------|------|
| 🟢 高 | 越界提交 = 0 OR 全部为合理越界（< 20 行 + 有 commit message 说明）|
| 🟡 中 | 1-2 处不合理越界（≥ 20 行 OR 无说明）|
| 🔴 低 | ≥ 3 处不合理越界 |

**特殊规则**：
- 项目未使用 flowsmith（无 arch.md）→ 此维度评分为 N/A，不参与展示

## 维度 4：经验吸收（Knowledge Absorption）

基于 `risks.md` 中"重蹈历史覆辙"章节：

| 评分 | 条件 |
|------|------|
| 🟢 高 | 0 处重蹈覆辙 |
| 🟡 中 | 1-2 处重蹈覆辙（非安全相关）|
| 🔴 低 | ≥ 1 处重蹈覆辙（安全相关）OR ≥ 3 处重蹈覆辙（任何类别）|

**特殊规则**：
- 项目未使用 flowsmith（无 lessons.md）→ 此维度评分为 N/A
- 重蹈覆辙的判定必须是"完美匹配"（部分匹配不计入）

## 评分输出强制要求

每条评分必须按以下格式：

```markdown
### {author_name}
- **完成度 🟢 高**：4/4 改动文件有测试 + 0 TODO + 0 调试遗留
  - 证据：commit a3f8c1d 同时修改源文件和测试文件
- **协作健康 🟢 高**：接口变更通知了所有调用方
  - 证据：UserRepository.findById 签名变更后，调用方在 commit b8e2f4a 同步更新
- **边界遵守 🟢 高**：100% 改动在 auth/ 模块内（plan.md 定义的范围）
- **经验吸收 🟢 高**：避免了 lessons.md 中的所有已知坑
```

**禁止**：
- "Alice 表现出色" — 评判性语言
- "比 Bob 更好" — 横向对比
- "可能存在问题" — 模糊不准的描述
- 没有证据的评分

## 评分使用边界（在 scores.md 中强制声明）

- 不应用于个人绩效考核
- 不应用于工资/晋升决策
- 不能反映：协作沟通质量、设计决策合理性、代码评审给出的反馈、跨任务知识传承
- 仅作为团队回顾的**参考起点**

## 项目无 flowsmith 上下文时的降级

若项目未使用 flowsmith（无 plan.md / arch.md / lessons.md）：
- 维度 1（完成度）：仍可评分，纯基于代码信号
- 维度 2（协作健康）：仍可评分，基于 git diff 分析
- 维度 3（边界遵守）：标注 N/A，不展示该列
- 维度 4（经验吸收）：标注 N/A，不展示该列

输出表格自动收缩，不展示 N/A 列。
