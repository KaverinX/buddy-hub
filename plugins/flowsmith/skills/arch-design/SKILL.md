# 架构设计（arch-design）

## 前置读取（每次执行必须先做）

1. 读取 `../task-planning/reference/state-machine.md` — 确认转换规则
2. 读取 `../task-planning/reference/document-schemas.md` — 获取 arch.md 格式
3. 读取 `.sop/state.json` — 确认 current_phase = ARCHITECTURE
4. 读取 `.sop/plan.md` — 理解任务目标、约束、风险

## 执行指令

你现在进入 **【阶段 2：架构设计（ARCHITECTURE）】**。

### Step 1 — 更新状态

将 `.sop/state.json` 中 `ARCHITECTURE.status` 改为 `"running"`。

### Step 2 — 架构深度思考（内部推导，不输出）

这是整个 SOP 中智识密度最高的阶段。在动笔前，完成以下思考：

**设计维度一：分解边界**
- 这个系统天然的"切割面"在哪里？（按业务域 / 按技术层 / 按变化频率）
- 高内聚低耦合：哪些东西应该在一起，哪些应该分开？
- 识别"变化的维度"：未来最可能变化的是哪部分？把它隔离。

**设计维度二：接口设计**
- 接口应该表达"做什么"，而不是"怎么做"（Tell, Don't Ask）
- 接口参数是否携带了调用方不该知道的实现细节？
- 考虑接口的演进性：如果需求变化，这个接口如何在不破坏调用方的前提下扩展？

**设计维度三：数据流设计**
- 数据在哪里被创建，在哪里被修改，在哪里被销毁？
- 状态是否集中管理？分散状态是复杂性的主要来源。
- 副作用（IO、网络、DB 写）是否被推到系统边界，而不是散落在核心逻辑中？

**设计维度四：错误路径设计（与正常路径同等重要）**
- 每个可能失败的操作，失败后系统处于什么状态？
- 操作是否具有幂等性？（重试安全）
- 是否需要事务边界？如果多步操作中途失败，如何回滚？
- 错误如何传播？是否应该在某个层边界做错误转换？

**设计维度五：非功能性要求**
- 并发：是否存在竞态条件？共享状态如何保护？
- 性能：热路径是什么？是否有 O(n²) 的操作隐藏在看似简单的循环里？
- 可观测性：关键路径上是否有足够的日志/指标可以 debug 生产问题？

### Step 3 — ADR 驱动的决策过程

对于每个关键设计决策，必须显式做 ADR，而不是把决策隐含在实现里。

**何为"关键"决策**：
- 选择了某种设计模式（而不是另一种）
- 引入了新的依赖
- 选择了特定的数据结构
- 在灵活性和简单性之间做了权衡
- 为了性能牺牲了代码可读性

ADR 不是事后补文档，是在做决策时帮助你想清楚为什么这样选。

### Step 4 — 生成 arch.md

严格按照 reference 中的 `arch.md` 格式生成文档。

**质量要求**：
- "整体设计思路"必须说清楚核心权衡，不接受"采用分层架构"这类无意义描述
- 每个模块的"禁止做的事"必须填写，边界不清的架构等于没有架构
- ADR 数量不少于 2 条，且每条必须有真正被考量过的备选方案（不接受只列一个方案的 ADR）
- "扩展性考量"必须填写，说明哪些设计决策是故意的简化（YAGNI）而非疏忽

### Step 5 — 更新状态

将 `.sop/state.json` 中 `ARCHITECTURE.status` 改为 `"done"`，附 completed_at。

```

```
